# ADR-007 — Двойное хранилище состояния: Redis + PostgreSQL

**Статус:** Принято
**Дата:** 2026-04-25
**Связанные ADR:** [ADR-001](ADR-001-microservices.md), [ADR-008](ADR-008-in-memory-job-store.md)

## Контекст

Историческая хронология:

1. На стадии MVP система была монолитом, и единственным хранилищем
   состояния выступал **PostgreSQL** (таблицы `processes`,
   `process_logs` — «ProcessRegistry»).
2. При переходе к микросервисной архитектуре (см.
   [ADR-001](ADR-001-microservices.md)) появилась потребность
   в shared state, доступном всем сервисам с минимальной латентностью —
   в первую очередь для **актуального статуса запуска**, который
   клиент опрашивает поллингом, а Gateway обновляет на каждой стадии
   пайплайна.
3. Хранить весь объём run-state в PostgreSQL оказалось нелогично:
   запросы статуса частые, RunRecord обновляется на каждом из 7 шагов
   пайплайна, а долгосрочно хранить эти горячие данные в реляционной БД
   избыточно.
4. С другой стороны, **выбрасывать всю историю запусков нельзя** —
   она нужна для аудита, отчётности по DPO, сравнения экспериментов
   во времени.

## Рассмотренные варианты

### A. Только PostgreSQL

Сохранить всё в PostgreSQL: `RunRecord` обновляется на каждом шаге
прямо в реляционной таблице.

* **За:** одно хранилище, нет дрейфа между ними; durable; ACID.
* **Против:** избыточная нагрузка на БД при частых апдейтах
  и поллинге; высокая латентность по сравнению с Redis;
  PostgreSQL — не оптимален как «горячий» кеш.

### B. Только Redis

Перенести в Redis всё, включая историю запусков.

* **За:** одно хранилище, низкая латентность для всех операций.
* **Против:** Redis — in-memory, durability требует RDB/AOF
  с компромиссами; долгосрочное хранение растущей истории за месяцы/годы
  в RAM нелогично; нет SQL для построения отчётности «процентов
  завершившихся вердиктом PASS за последний квартал».

### C. Redis (горячее) + PostgreSQL (холодное)

Redis хранит только активные / недавние запуски с TTL 1 час;
PostgreSQL хранит долгосрочную историю в `processes`. Запрос
«все запуски» сливает оба источника, отдавая приоритет Redis
(он содержит больше деталей).

* **За:** каждое хранилище используется по своему профилю; быстрый
  поллинг, durable история; чёткое разделение «hot path / cold path».
* **Против:** дрейф между ними возможен; нужна merge-логика
  в Gateway; больше операционной сложности.

### D. PostgreSQL + UNLOGGED tables / Redis-кеш поверх

Кешировать запросы статуса в Redis перед PostgreSQL.

* **За:** PostgreSQL остаётся single source of truth.
* **Против:** усложняет write-path (cache invalidation); все апдейты
  всё равно бьют в PostgreSQL.

## Решение

Принят вариант **C — Redis (горячее) + PostgreSQL (холодное)**.

Реализация:

* **Redis (`final_system/api/store.py:RunStore`):**
  * Ключ `run:{run_id}` — JSON-сериализованный `RunRecord`.
  * `ZSET runs:by_created` — сортировка по дате создания.
  * Optimistic locking через `WATCH/MULTI/EXEC` (до 3 retry).
  * **TTL 1 час** после `failed` (для `completed` TTL пока
    не выставляется в текущем коде; завершённые записи остаются
    до явного удаления через `DELETE /runs/{id}` либо чистятся
    отдельно).
* **PostgreSQL (`final_system/db/init_db.sql` → таблица `processes`):**
  * Долгосрочная история запусков.
  * Используется как источник для `GET /runs` после истечения TTL
    из Redis.
  * Disable-able через `DB_DISABLED=true` — Gateway работает
    и без неё в режиме «только Redis».

* **API:**
  * `GET /runs/active` — **только Redis**, активные и недавние запуски.
  * `GET /runs` — **Redis + PostgreSQL**, объединение с приоритетом
    Redis при пересечении `run_id`.

Аргументация:

1. Каждое хранилище используется по своему профилю: Redis закрывает
   hot-path (частый поллинг, частые апдейты), PostgreSQL —
   cold-path (история, аудит, отчётность).
2. Сценарий «клиент опрашивает статус каждые 5 секунд» не нагружает
   PostgreSQL.
3. PostgreSQL изначально был частью MVP-монолита — переиспользование
   уже работающей `processes`-таблицы дешевле, чем выкидывать её
   и перетаскивать всё в Redis.
4. Поведение endpoint'ов прозрачно для клиента: `/runs` — «всё»,
   `/runs/active` — «свежее», что соответствует двум типичным
   сценариям использования (мониторинг текущего vs аудит истории).

## Последствия

### Положительные

* Поллинг статуса (`GET /runs/{id}` и `GET /runs/active`) обслуживается
  Redis с минимальной латентностью и без нагрузки на PostgreSQL.
* Optimistic locking в Redis (`WATCH/MULTI/EXEC`) защищает от race
  conditions при конкурентных апдейтах одного `RunRecord`.
* PostgreSQL остаётся durable-хранилищем истории, доступным для
  SQL-запросов отчётности.
* Сервис продолжает работать при недоступности PostgreSQL
  (`_list_pg_runs` возвращает пустой список при ошибке, не валит API).
* Можно полностью отключить PostgreSQL через `DB_DISABLED=true` —
  упрощает локальную разработку.

### Отрицательные

* **Дрейф данных.** Поля в Redis-`RunRecord` богаче, чем в PostgreSQL-
  `processes`; merge-логика в `routers/runs.py:list_all_runs`
  отдаёт приоритет Redis, но за пределами TTL информация может быть
  усечённой.
* **Дублирование апдейтов.** При завершении пайплайна нужно писать
  и в Redis (для активного состояния), и в PostgreSQL (для истории).
  В текущей реализации запись в PostgreSQL ProcessRegistry — отдельный
  слой; синхронизация — обязанность Gateway.
* **Разные модели данных.** RunRecord (Redis) и processes (PostgreSQL)
  имеют разные поля и формат — `from_pg_row()` и `from_record()`
  выполняют конвертацию. Любое изменение схемы требует правок
  в обоих местах.
* Нужны два отдельных healthcheck (`/health` для Redis уровня
  docker-compose, `/health/db` для PostgreSQL).

### Условия пересмотра

* Если ProcessRegistry перестанет использоваться (например, история
  будет вынесена в data lake или централизованную observability-систему) —
  PostgreSQL можно вывести из системы, оставив только Redis.
* Если, наоборот, появится требование durable-хранения всех run-state
  (вне зависимости от TTL) — придётся либо увеличивать TTL Redis,
  либо вводить background-flush из Redis в PostgreSQL.
