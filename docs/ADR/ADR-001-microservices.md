# ADR-001 — Микросервисная архитектура (5 сервисов)

**Статус:** Принято
**Дата:** 2026-04-25
**Связанные ADR:** [ADR-002](ADR-002-docker-compose.md), [ADR-003](ADR-003-shared-volume.md), [ADR-011](ADR-011-inline-generator-config.md), [ADR-012](ADR-012-lazy-ml-imports.md)

## Контекст

Система реализует пайплайн «импорт → препроцессинг → синтез → оценка
приватности и полезности → отчёт». На стадии MVP это был монолитный
скрипт (`run_pipeline()` вызывал по очереди модули). При переходе
к сервисной форме встал вопрос: оставить монолит за HTTP-фасадом или
разбить на отдельные микросервисы — и если разбивать, то на сколько.

Дополнительные ограничения, влияющие на выбор:

* Целевая аудитория — крупные компании, которые могут захотеть
  **подменить отдельные стадии** своими реализациями (например,
  использовать собственный набор оценочных метрик, удовлетворяющий
  внутренним стандартам безопасности).
* Профиль зависимостей сильно неравномерный: стадия синтеза тянет
  `torch`, `opacus`, `smartnoise-synth`, `sdv` (~13 ГБ образ),
  стадия отчётности — практически ничего, кроме `fastapi`+`pydantic`.
* Шаги пайплайна напрямую соответствуют доменам предварительного
  национального стандарта (ПНСТ) по синтетическим данным:
  «генерация», «оценка полезности», «оценка приватности»,
  «формальные гарантии», «отчётность».

## Рассмотренные варианты

### A. Монолит за HTTP-фасадом (один контейнер)

Один процесс, один Docker-образ, все стадии вызываются как функции
Python внутри одного запроса (или фонового потока).

* **За:** простейший деплой; нет сетевых хопов; нет проблем с разделяемым
  стораджем.
* **Против:** один Docker-образ ~13 ГБ для всех; невозможно подменить
  одну стадию без перебилда всего; командам пришлось бы ставить
  тяжёлые ML-зависимости даже для разработки HTTP-фасада.

### B. Two-tier (API + один heavy worker)

Один публичный API + один «worker» сервис, в котором живёт всё ML.

* **За:** изоляция ML-зависимостей от публичного API.
* **Против:** worker всё равно остаётся монолитом — заменить
  «только evaluation» нельзя без затрагивания synthesis.

### C. Микросервисы по логическим доменам (5 сервисов)

Gateway + Data + Synthesis + Evaluation + Reporting. Каждый шаг ПНСТ
(кроме дублирующих по логике, как импорт-экспорт) — отдельный сервис.

* **За:** возможность подменить отдельный сервис у заказчика; чёткое
  соответствие шагам стандарта; минимальная связность по зависимостям
  (Reporting не знает про torch).
* **Против:** больше операционной сложности; сетевые хопы; необходим
  shared storage для артефактов; разделяемые контракты надо поддерживать.

### D. «Атомарные» микросервисы (10+ сервисов)

Дальнейшее дробление: отдельный сервис на classical-метрики, на DCR,
на TSTR/TRTR, и т.д.

* **За:** максимальная гибкость подмены.
* **Против:** оверинжиниринг для проекта такого scope; кратный рост
  накладных расходов; дробление без чёткого доменного смысла.

## Решение

Принят вариант **C — пять микросервисов**:

| Сервис | Порт | Ответственность |
|---|---|---|
| Gateway (`app`) | 8000 | HTTP-вход, оркестрация, состояние run |
| Data Service | 8001 | импорт, препроцессинг, holdout split |
| Synthesis Service | 8002 | обучение генераторов, сэмплирование |
| Evaluation Service | 8003 | privacy + utility метрики |
| Reporting Service | 8004 | сборка отчёта и вердикт PASS/FAIL/PARTIAL |

Принципы разбиения:

1. **Логическое разделение доменов.** Каждый сервис — отдельный шаг
   доменного пайплайна, понятный без знания реализации.
2. **Заменяемость для крупных заказчиков.** Внутренняя команда
   заказчика может реализовать собственный сервис, удовлетворяющий
   тому же контракту (например, свои метрики приватности),
   и подключить его вместо стандартного без правок других сервисов.
3. **Соответствие ПНСТ.** Один шаг стандарта = один сервис, кроме
   случаев, когда шаги дублируют друг друга по логике (импорт и экспорт
   данных живут внутри Data Service / Gateway, а не в отдельном сервисе).

Evaluation и Reporting сохранены отдельными несмотря на лёгкость обоих —
они принадлежат разным доменам стандарта (оценка vs формирование
отчётности и вынесение вердикта), и заказчик может захотеть подменить
один без другого (например, оставить наши метрики, но писать отчёт
в собственном корпоративном формате).

## Последствия

### Положительные

* Любой из пяти сервисов можно переписать или подменить, сохранив
  контракт (определён в `final_system/shared/schemas/`) — остальные
  четыре остаются нетронутыми.
* Тяжёлые ML-зависимости изолированы в одном образе; Gateway-образ
  и Reporting-образ остаются слим (десятки–сотни МБ).
* При отладке падение в стадии видно по логам конкретного сервиса
  без необходимости разбираться в одном большом stack trace.
* Архитектура напрямую отражает шаги ПНСТ — упрощает аудит соответствия
  стандарту.

### Отрицательные

* Сетевые вызовы между сервисами вместо вызовов функций — повышенная
  латентность и больше точек отказа.
* Контракты `shared/schemas/` нужно поддерживать в синхронизации
  с реализациями обоих концов.
* Артефакты передаются не телом HTTP, а через разделяемый storage
  (см. [ADR-003](ADR-003-shared-volume.md)) — это вводит ограничение
  single-host для текущего MVP.
* Развёртывание сложнее: нужен docker-compose / kubernetes / иной
  оркестратор вместо `python -m`.

### Условия пересмотра

Решение остаётся в силе. Возможные триггеры частичного пересмотра
(не отказа от микросервисов как таковых):

* Если выяснится, что заказчики подменяют Evaluation и Reporting
  всегда вместе — их стоит объединить в один сервис «Post-processing».
* Если появится синтетика реляционных таблиц / временных рядов
  с принципиально другим pipeline — может потребоваться шестой
  сервис либо вынос multi-table-логики в Data Service.
