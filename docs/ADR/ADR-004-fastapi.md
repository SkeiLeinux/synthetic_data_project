# ADR-004 — FastAPI как HTTP-фреймворк

**Статус:** Принято
**Дата:** 2026-04-25
**Связанные ADR:** [ADR-005](ADR-005-pydantic.md), [ADR-010](ADR-010-background-tasks.md)

## Контекст

Все 5 сервисов системы (Gateway + 4 микросервиса) — это HTTP-API.
Нужно выбрать веб-фреймворк для всех пяти.

Жёсткие ограничения, заданные природой проекта:

* **Язык — Python.** Вся ML-инфраструктура (`torch`, `opacus`,
  `smartnoise-synth`, `sdv`, `ctgan`, `scikit-learn`, `pandas`)
  существует только в Python (или с Python как первого класса).
  Альтернативный язык на бэкенде сделал бы невозможным переиспользование
  кода между API и ML-логикой и потребовал бы введения дополнительной
  сериализационной границы внутри одного сервиса.
* **Архитектурный стиль — REST.** REST — наиболее распространённый
  и наиболее знакомый автору стиль API; команды разработки и SDK на
  стороне заказчиков с REST работают в первую очередь; OpenAPI как
  стандарт документации развился именно вокруг REST.

При двух этих ограничениях пространство выбора сужается до Python REST-
фреймворков.

## Рассмотренные варианты

### A. FastAPI

Современный async-first фреймворк, нативная Pydantic-интеграция,
автогенерация OpenAPI 3.x + Swagger UI, dependency injection.

* **За:** автоматический OpenAPI/Swagger без отдельной декларации;
  нативная Pydantic v2-валидация — один и тот же класс описывает
  и контракт, и сериализацию, и runtime-валидацию (см.
  [ADR-005](ADR-005-pydantic.md)); встроенные `BackgroundTasks`
  (см. [ADR-010](ADR-010-background-tasks.md)); dependency injection
  для разрешения авторизации, settings, клиентов; активное сообщество.
* **Против:** относительно молодой по сравнению с Flask/Django —
  меньше готовых интеграций для редких случаев; async-first может
  быть избыточен в полностью синхронных пайплайнах.

### B. Flask + Flask-RESTX / Flask-Smorest

Классический микрофреймворк с REST-расширениями.

* **За:** зрелость, огромная экосистема, много готовых рецептов.
* **Против:** OpenAPI-генерация — не нативная, требует расширений
  с собственными конвенциями; Pydantic нужно подключать вручную;
  валидация и сериализация — отдельные шаги; нет нативного async
  (требует gevent/quart).

### C. Django REST Framework

Полноценный фреймворк с ORM, admin, миграциями.

* **За:** богатый набор готовой инфраструктуры (auth, admin, ORM).
* **Против:** избыточен для API-only сервисов; своя Models/Serializers
  парадигма, не Pydantic; вес фреймворка несоразмерен задаче;
  настройки Django под микросервис — отдельная боль.

### D. Litestar / Starlette / aiohttp / Sanic

Прочие async-фреймворки.

* **За:** разные нюансы производительности и API-стиля.
* **Против:** меньше комьюнити, чем у FastAPI; меньше готовых рецептов
  для типовых задач; для целей проекта не дают ничего, чего не даёт
  FastAPI.

## Решение

Принят вариант **A — FastAPI**.

Выбор сделан без детального бенчмарк-сравнения с альтернативами,
поскольку при ограничениях «Python + REST» FastAPI — отраслевой
стандарт де-факто на 2024–2026 год для проектов со следующим профилем:

1. **API + ML.** Нужна простая интеграция Python ML-библиотек
   с HTTP-фасадом — ровно то, ради чего FastAPI создавался.
2. **Контракты как код.** Pydantic-схемы используются и для
   валидации, и для сериализации, и для генерации OpenAPI —
   один источник правды (см. [ADR-005](ADR-005-pydantic.md)).
3. **Автодокументация.** `/docs` (Swagger UI) и `/openapi.json`
   доступны из коробки на каждом сервисе — снижает порог входа
   для интеграторов.
4. **Минимальная boilerplate.** Декораторы маршрутов, типизация
   параметров через type hints, dependency injection через `Depends`
   делают код компактным.

## Последствия

### Положительные

* Pydantic-схемы из `final_system/shared/schemas/` валидируются
  фреймворком автоматически — нет отдельного слоя `validate()`.
* Для каждого сервиса доступны живые `/docs` (Swagger UI)
  и `/openapi.json` без дополнительной настройки. Параллельно
  поддерживаются «офлайн» спеки в директории `openapi/`.
* `Depends(require_auth)` и `Depends(get_settings)` дают единый
  механизм для авторизации и настроек.
* `BackgroundTasks` позволяет реализовать асинхронный пайплайн
  без отдельного worker-сервиса (см.
  [ADR-010](ADR-010-background-tasks.md)).
* Type-hints в сигнатурах ручек выступают одновременно
  документацией и валидацией — низкий риск рассинхронизации.

### Отрицательные

* Vendor-lock на FastAPI: миграция на другой Python-фреймворк
  потребует перехода с `Depends`-based DI и от частей Pydantic-API.
* Async-семантика — лёгкий источник ошибок в синхронном коде
  (приходится явно писать `def`, а не `async def`, чтобы
  фреймворк выполнил функцию в threadpool).
* Авторизация на уровне приложения (`Depends(require_auth)`)
  ограничена возможностями FastAPI — для production придётся
  вынести часть auth-логики на API Gateway / service mesh.

### Условия пересмотра

В обозримом будущем — **не пересматривать**. Возможные триггеры
теоретического пересмотра:

* Кардинальное изменение языка (например, переезд части системы на Go),
  что само по себе потребует радикального reanalysis всей архитектуры.
* Появление принципиально лучшего Python REST-фреймворка с массовым
  переходом сообщества (на момент принятия решения такового нет).
