# ADR-015 — Holdout split в Data Service до синтеза

**Статус:** Принято
**Дата:** 2026-04-25
**Связанные ADR:** [ADR-001](ADR-001-microservices.md)

## Контекст

Для оценки синтетических данных необходима **отложенная выборка
(holdout)** — часть реальных данных, которую генератор **не видел
при обучении**. Holdout используется в:

* **Utility (TSTR / TRTR).** TRTR обучается на real_train, тестируется
  на holdout (потолок ML-качества). TSTR обучается на синтетике,
  тестируется на том же holdout (качество синтетики). Utility Loss =
  TRTR − TSTR — корректно только при общем holdout.
* **Privacy (DCR_holdout, NNDR, MIA).** DCR от синтетики до train
  сравнивается с DCR от holdout до train — если они близки,
  меморизации нет. MIA-классификатор различает train (label=1)
  и holdout (label=0) по их близости к синтетике.

**Главная инвариант:** генератор **никогда** не должен видеть holdout
ни на каком этапе. Если эта инвариант нарушается — все метрики
приватности (особенно MIA) теряют смысл.

Вопрос: где в архитектуре сделать split?

## Рассмотренные варианты

### A. Split внутри каждого эвалюатора

Каждый сервис оценки (`evaluation_service`) выполняет свой split
прямо перед расчётом метрик.

* **За:** локально, не нужно передавать `split_id` между сервисами.
* **Против:** **критическая методологическая ошибка** — каждый
  эвалюатор может получить разный holdout (если random_state
  разный) или одинаковый, но генератор всё равно мог обучиться
  на полных данных (включая будущий holdout); единая инвариант
  «генератор не видит holdout» теряется.

### B. Split в Synthesis Service перед обучением

Synthesis Service сам отделяет holdout, обучается на оставшемся,
возвращает holdout наружу.

* **За:** генератор гарантированно не видит holdout (split делает
  тот же сервис).
* **Против:** Synthesis Service берёт на себя ответственность
  за preprocessing и сплит — выходит за его доменную область
  (его задача — обучать); все downstream-сервисы должны как-то
  получить тот же holdout, что неудобно при двух сервисах оценки.

### C. Split в Data Service атомарно с preprocessing

Data Service выполняет preprocessing (дубликаты, пропуски,
минимизация) и сразу же — stratified holdout split. Сохраняет
два файла `train.csv` и `holdout.csv` под общим `split_id`.
Все последующие сервисы (Synthesis, Evaluation) ссылаются на
этот `split_id` — гарантия одинакового holdout у всех.

* **За:** инвариант «генератор не видит holdout» обеспечивается
  архитектурно (holdout попадает только в Evaluation Service);
  атомарность: preprocessing и сплит делаются один раз; все
  downstream-сервисы видят один и тот же holdout.
* **Против:** одноразовость — нельзя пересплитить с другим
  random_state без создания нового `split_id`.

### D. Split на стороне Gateway

Gateway сам читает CSV, делает split, отправляет train в Synthesis,
holdout в Evaluation отдельно.

* **За:** Gateway имеет полный контроль.
* **Против:** Gateway тащит pandas + sklearn в горячий путь;
  большие CSV циркулируют через Gateway-память; Gateway теряет
  свой статус «тонкого оркестратора»; противоречит
  [ADR-012](ADR-012-lazy-ml-imports.md) (нужен `train_test_split`
  и работа с DataFrame).

## Решение

Принят вариант **C — split в Data Service атомарно с preprocessing**.

Реализация — `final_system/services/data_service/router.py:split_dataset`:

* Эндпоинт `POST /api/v1/datasets/{dataset_id}/split` принимает
  параметры split'а (`holdout_size`, `random_state`, `target_column`,
  ...) и **выполняет всё за один вызов**:
  1. (опц.) sampling датасета до `sample_size`;
  2. preprocessing (дубликаты, пропуски);
  3. minimization (удаление direct identifiers, опц. high-cardinality);
  4. detect / override типы колонок;
  5. stratified `train_test_split` (sklearn) с `stratify=target_column`
     если задан;
  6. атомарное сохранение `splits/{split_id}/train.csv`,
     `splits/{split_id}/holdout.csv`, `profile.json`, `meta.json`.
* Все downstream-сервисы (Synthesis, Evaluation) принимают на вход
  `split_id` и читают из общего расположения на shared volume
  (см. [ADR-003](ADR-003-shared-volume.md)).
* Synthesis Service читает только `train.csv` (и не имеет повода
  читать `holdout.csv`).
* Evaluation Service читает оба файла + `synthetic.csv`, выполняет
  privacy и utility оценку.

Аргументация:

1. **Архитектурная гарантия инварианта.** Holdout физически
   попадает только в Evaluation Service. Synthesis Service
   читает другой файл (`train.csv`) — даже если в его коде
   будет ошибка, он не сможет случайно обучиться на holdout.
2. **Один и тот же holdout у всех потребителей.** Privacy и Utility
   эвалюаторы используют одинаковый `split_id` → одинаковый
   `holdout.csv` → их результаты сопоставимы.
3. **Атомарность.** Если бы preprocessing и split были разделены,
   получали бы две точки, между которыми можно «подменить» данные.
   Атомарный шаг — одна транзакция: данные обработаны и разделены
   за один вызов.
4. **Доменное соответствие.** Data Service отвечает за «подготовку
   данных» — split логически принадлежит этому домену
   (см. [ADR-001](ADR-001-microservices.md)).
5. **Стратификация.** При задании `target_column` `train_test_split`
   стратифицирует разбиение по нему — баланс классов сохраняется
   и в train, и в holdout. Это нельзя сделать одинаково в двух
   независимых сплитах с разной seed.

## Последствия

### Положительные

* **Методологически корректная оценка приватности.** Гарантия
  «holdout невидим генератору» обеспечивается архитектурно,
  а не дисциплиной разработчиков.
* **Единый holdout** для всех сервисов оценки — Privacy и Utility
  возвращают сопоставимые цифры.
* Профиль предобработанного датасета (`profile.json`) сохраняется
  один раз на одном объединённом DataFrame (до сплита) —
  отражает реальное состояние данных, на которых будет работать
  пайплайн.
* Параметры split'а аудитируются: `random_state`, `holdout_size`,
  `force_categorical`, `direct_identifiers` сохранены в `SplitMeta`
  (`splits/{split_id}/meta.json`).

### Отрицательные

* **Одноразовость.** Если хочется пересчитать оценку с другим
  random_state или другой долей holdout — нужно создавать новый
  `split_id`. Старый split остаётся как артефакт.
* **Привязка к одному формату holdout.** Если в будущем понадобится
  k-fold валидация, time-based split или иной не-stratified-split,
  потребуется расширение API сплита.
* Все downstream-сервисы должны соблюдать договорённость
  «читай по `split_id`» — если кто-то самостоятельно сделает
  свой split, инвариант сломается. Архитектурно это сейчас
  обеспечивается тем, что Synthesis Service просто не получает
  holdout (его API не предусматривает этого), а Evaluation Service
  ожидает `split_id` в каждом запросе.

### Условия пересмотра

Возможные триггеры частичного пересмотра:

* Появление требования k-fold cross-validation для ML-эффективности —
  потребует расширения `SplitMeta` (несколько train/test пар вместо
  одной).
* Появление временных рядов / последовательных данных — потребует
  time-based split вместо random.
* Multi-tenancy с разными политиками split'а у разных команд —
  возможно, потребуется параметризация по политикам.

Сама архитектурная инвариант «split — атомарно, до синтеза, в
одном месте» сохранится при любом из этих изменений.
