# ADR-005 — Pydantic v2 как контракт между сервисами

**Статус:** Принято
**Дата:** 2026-04-25
**Связанные ADR:** [ADR-001](ADR-001-microservices.md), [ADR-004](ADR-004-fastapi.md), [ADR-011](ADR-011-inline-generator-config.md)

## Контекст

Микросервисы (см. [ADR-001](ADR-001-microservices.md)) обмениваются
JSON-сообщениями по HTTP. Нужно выбрать формат описания контрактов:

* **Запросов и ответов** между сервисами (например, `SynthesisJobCreate`,
  `PrivacyEvalRequest`, `ReportRequest`).
* **Внутренних доменных моделей** (`RunRecord`, `JobRecord`, `DataSchema`).
* **Конфигурации** пайплайна (`AppConfig`, `GeneratorYamlConfig`,
  `ThresholdsYamlConfig` в `config_loader.py`).

Контракт должен:

1. Валидироваться в runtime (типы, диапазоны, обязательность полей).
2. Сериализоваться в JSON и обратно.
3. Документироваться машинно-читаемо (для генерации OpenAPI).
4. Жить в одном месте — `final_system/shared/schemas/` —
   и переиспользоваться обоими концами вызова.

## Рассмотренные варианты

### A. Pydantic v2

Декларативные классы с типизированными полями; рантайм-валидация;
сериализация в JSON; нативная интеграция с FastAPI и автогенерация
OpenAPI.

* **За:** один класс закрывает все четыре задачи; экосистема FastAPI;
  v2 — переписан на Rust, существенно быстрее v1; `model_validate`,
  `model_dump`, `model_dump_json` — компактный API.
* **Против:** Python-only — если в будущем появится сервис на другом
  языке, контракт придётся транслировать вручную.

### B. dataclasses + jsonschema (или собственная валидация)

Стандартные Python `@dataclass` для типизации + сторонний
`jsonschema` для валидации.

* **За:** только стандартная библиотека для типов; нет vendor-lock.
* **Против:** валидация и сериализация — отдельные шаги;
  нужно либо генерировать JSON Schema из dataclasses вручную,
  либо поддерживать две декларации; конвертация в JSON
  и обратно — boilerplate.

### C. attrs / cattrs

`attrs` для классов + `cattrs` для (де)сериализации.

* **За:** более гибкая, чем dataclasses; проверенная в больших проектах.
* **Против:** не интегрируется нативно с FastAPI — потребуется
  собственный слой; меньше готовых конвертеров для типовых случаев.

### D. Protobuf / gRPC contracts

Контракты в `.proto`-файлах с кодогенерацией Python-классов.

* **За:** мульти-язычность из коробки; компактная бинарная
  сериализация; чёткое разделение схемы и кода.
* **Против:** REST-API становится двухуровневой кодогенерацией
  (proto → Python + ручное связывание с FastAPI-handler'ами);
  для проекта, целиком на Python, добавляет сложность без
  компенсирующей выгоды; OpenAPI всё равно придётся генерировать
  отдельно.

### E. Marshmallow

Классическая Python-библиотека для (де)сериализации с валидацией.

* **За:** зрелость, проверенность.
* **Против:** Schemas декларируются отдельно от моделей —
  два места, которые надо синхронизировать; типизация через
  поля `fields.String()` менее «питонична», чем type hints;
  вне FastAPI-экосистемы.

## Решение

Принят вариант **A — Pydantic v2**.

Применение:

* **Контракты HTTP** — все Pydantic-классы в `final_system/shared/schemas/`
  (`datasets.py`, `synthesis.py`, `evaluation.py`, `reporting.py`)
  и `final_system/api/schemas/`.
* **Конфиги YAML** — `final_system/config_loader.py` (`AppConfig`,
  `GeneratorYamlConfig`, `PipelineConfig`, `ThresholdsYamlConfig`,
  и т.д.); валидация при загрузке конфига отлавливает ошибки
  на старте, а не в середине обучения.
* **Доменные модели runtime-state** — `RunRecord`, `JobRecord` —
  как `@dataclass` (т.к. они не сериализуются по HTTP, а только
  локально хранятся в Redis/in-memory).
* **Settings** — `pydantic-settings` для всех `Settings` классов
  на каждом сервисе.

Аргументация:

1. Pydantic-классы выступают **единым источником правды** для
   контракта: типы, валидация, сериализация, OpenAPI-описание —
   из одной декларации.
2. FastAPI (см. [ADR-004](ADR-004-fastapi.md)) использует Pydantic
   нативно для парсинга `request body`, валидации параметров и
   генерации `/openapi.json` — нет дублирования логики.
3. Field validators (`@field_validator`) и model validators
   (`@model_validator`) позволяют явно описать инварианты домена
   (например, в `GeneratorYamlConfig` проверка `preprocessor_eps < epsilon`),
   что делает их частью контракта, а не разрозненными ассертами.
4. Pydantic v2 переписан на Rust — производительность достаточна
   даже для парсинга крупных JSON-отчётов на стороне Reporting Service.

## Последствия

### Положительные

* Один файл схемы (`shared/schemas/synthesis.py`) описывает контракт,
  по которому Gateway и Synthesis Service общаются.
* Невалидные запросы отбиваются до попадания в код handler'а
  с подробным сообщением об ошибке (имя поля, тип, значение).
* Любые изменения контракта (новое поле, переименование)
  проявляются как ошибки валидации сразу, а не как «странные
  значения» в середине пайплайна.
* `@field_validator` в `GeneratorYamlConfig` ловит логические ошибки
  конфига (epsilon ≤ 0, holdout_size вне (0, 1), preprocessor_eps ≥ epsilon)
  на этапе загрузки YAML — задолго до запуска синтеза.
* Бесплатный `/openapi.json` со всеми схемами — снижает
  трудозатраты на документацию.

### Отрицательные

* **Python-only.** Если когда-нибудь появится сервис на другом
  языке, его придётся либо генерировать клиент через openapi-generator,
  либо дублировать схемы вручную.
* Pydantic v2 поломал часть API v1 — миграция при обновлении
  крупных проектов потребовала усилий; в этом проекте использован
  v2 с самого начала, риски миграции уже отыграны.
* Pydantic-классы плотно завязаны на Python type-hints —
  любое отклонение от стандартной типизации (например, динамические
  поля) требует `model_config` overrides.

### Условия пересмотра

Не пересматривать в обозримом будущем. Триггеры теоретического пересмотра:

* Появление политики «контракты должны жить в proto/IDL» при интеграции
  в крупную корпоративную платформу контрактов.
* Появление сервисов на не-Python языках, где надо явно использовать
  один и тот же контракт.

В обоих случаях пересмотр будет точечным (контракты HTTP), а Pydantic
останется для конфигов и settings.
