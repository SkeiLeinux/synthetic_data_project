# ADR-010 — `BackgroundTasks` FastAPI вместо Celery / RQ для пайплайна

**Статус:** Принято (с известным техническим долгом)
**Дата:** 2026-04-25
**Связанные ADR:** [ADR-001](ADR-001-microservices.md), [ADR-004](ADR-004-fastapi.md), [ADR-008](ADR-008-in-memory-job-store.md)

## Контекст

`POST /api/v1/runs` запускает длительный пайплайн (минуты — часы):
data import → split → synthesis → evaluation → reporting. Клиент
не должен ждать завершения в одном HTTP-запросе. Нужен механизм:

* немедленно вернуть `202 Accepted` + `run_id`;
* выполнить пайплайн в фоне;
* сохранить результат в RunStore (Redis), при необходимости — отправить
  webhook.

## Рассмотренные варианты

### A. FastAPI `BackgroundTasks`

Передать функцию `_execute_pipeline` в `BackgroundTasks.add_task()`;
FastAPI выполнит её после возврата ответа клиенту, в том же
процессе.

* **За:** ноль дополнительных зависимостей; код пайплайна
  тривиально вызывается как функция; для single-worker single-replica
  деплоя достаточно.
* **Против:** падение/рестарт Gateway убивает все активные пайплайны;
  невозможно горизонтально масштабировать; нет retry-policy;
  нет dead-letter queue; нагрузка на Gateway растёт с числом
  параллельных запусков.

### B. Celery + Redis (broker) + Redis (backend)

Полноценная очередь задач: Gateway пушит таску в Celery,
отдельный пул воркеров её выполняет.

* **За:** durable очередь; multi-replica воркеров; retry, ETA,
  rate limiting; стандарт Python-экосистемы.
* **Против:** существенный операционный оверхед (broker, backend,
  beat-процесс, отдельные worker-контейнеры, мониторинг flower);
  дублирование уже имеющегося Redis с разными use case'ами;
  нагрузка/настройка несоразмерна одному типу долгой задачи.

### C. RQ / Dramatiq

Более лёгкие альтернативы Celery.

* **За:** проще настроить, чем Celery.
* **Против:** всё равно требуют отдельного worker-контейнера и
  broker'а (RQ — Redis); для одного типа задачи в MVP избыточно.

### D. Temporal.io / Apache Airflow / Prefect

Workflow-движки с явными шагами и checkpoint'ами.

* **За:** durable workflow, recoverable из любой точки;
  хорошо для длинных конвейеров.
* **Против:** очень тяжёлый стек для MVP; время на освоение и
  настройку — сопоставимо с самим написанием системы.

### E. Threading через `threading.Thread(target=...)` в Gateway

Самый минимальный вариант — поднять daemon-поток вручную.

* **За:** нулевой оверхед.
* **Против:** ручной life-cycle; FastAPI `BackgroundTasks` делает
  то же самое и лучше интегрируется с request/response cycle.

## Решение

Принят вариант **A — FastAPI `BackgroundTasks`**.

Реализация — `final_system/api/routers/runs.py:create_run`:

```python
@router.post("", status_code=202)
def create_run(body, background_tasks, settings, _):
    run_store.add(record)
    background_tasks.add_task(_execute_pipeline, run_id=run_id, ...)
    return RunSummary.from_record(record)
```

`_execute_pipeline` — синхронная функция (выполняется в threadpool
FastAPI), которая последовательно вызывает 4 микросервиса через
синхронный httpx-клиент и обновляет `RunRecord` в Redis на каждом
шаге.

Аргументация:

1. **Соответствие профилю нагрузки MVP.** Активных пайплайнов
   одновременно — единицы; нет потока в сотни/тысячи задач/час.
2. **Минимум зависимостей.** Не вводится broker, не появляется
   отдельный worker-контейнер; стек остаётся компактным
   (см. [ADR-002](ADR-002-docker-compose.md)).
3. **Одинаковый паттерн с
   [ADR-008](ADR-008-in-memory-job-store.md).** Состояние пайплайна
   живёт там же, где Gateway — в Redis (`RunStore`); состояние
   синтез-джоба живёт там же, где Synthesis Service (in-memory).
4. **Технический долг признан явно.** Решение вынесено как «MVP-
   достаточный, требует замены при production» — это зафиксировано
   в PRD (раздел 12.1.1) и в текущем ADR.

Реальная работа делегируется микросервисам (Synthesis обучает
часами, Gateway просто поллит) — `BackgroundTasks` Gateway
выполняет лёгкую оркестрацию, а не само ML.

## Последствия

### Положительные

* Минимальный код для асинхронного запуска (одна строка
  `background_tasks.add_task(...)`).
* Нет дополнительных контейнеров (broker / worker / beat).
* Стек запускается одной командой `docker compose up -d --build`,
  без отдельной инициализации очереди.
* Отмена пайплайна (`DELETE /runs/{id}`) реализуется через
  обновление статуса в Redis + cancel джоба синтеза — без
  работы с broker-protocol.

### Отрицательные

* **Рестарт Gateway убивает все активные пайплайны.** Текущее
  поведение: `_execute_pipeline` падает с исключением; запуск
  остаётся в Redis в статусе `running` без обновлений и истекает
  по TTL. На стороне клиента это выглядит как «зависший» запуск.
* **Невозможно горизонтально масштабировать Gateway.** Если бы
  захотели две реплики — пришлось бы решать, какая реплика
  обрабатывает какой `run_id`; `BackgroundTasks` локальны
  для процесса.
* **Нет встроенного retry.** Если один из четырёх микросервисов
  ответил 5xx — `_execute_pipeline` поймает исключение, пометит
  run как `failed`. Никаких автоматических повторов нет.
* **Нет priority / rate limiting.** Пять одновременных
  `POST /runs` запустят 5 параллельных пайплайнов в одном
  Gateway-процессе — ограничено только числом потоков uvicorn.
* Длительная нагрузка `BackgroundTasks` повышает risk что
  health-check Gateway замедлится.

### Условия пересмотра

Перейти на полноценную очередь (PRD 12.1.1) при наступлении
любого из:

* Появление требования multi-replica Gateway.
* Появление требования durability пайплайна между рестартами
  (рестарт Gateway не должен убивать обучение, занимающее часы).
* Появление потока больше нескольких параллельных запусков
  (одной машине не хватает).
* Требование retry-policy и dead-letter queue.

Кандидаты на замену перечислены в
[ADR-008](ADR-008-in-memory-job-store.md) — вероятно,
единое решение для обоих долгов: один менеджер очередей,
обслуживающий и пайплайн-задачи Gateway, и джобы Synthesis Service.
