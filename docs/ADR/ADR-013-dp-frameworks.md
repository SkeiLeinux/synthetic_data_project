# ADR-013 — SmartNoise (DP-CTGAN) + Opacus (DP-TVAE) как DP-фреймворки

**Статус:** Принято
**Дата:** 2026-04-25
**Связанные ADR:** [ADR-001](ADR-001-microservices.md)

## Контекст

Один из ключевых функциональных требований — **формальные DP-гарантии**
для сгенерированных данных (см. PRD, FR-03.1, FR-03.8). Нужны
табличные генеративные модели с встроенным DP-SGD (или эквивалентным
DP-механизмом) и privacy accountant'ом, считающим фактически
потраченный ε.

На MVP-стадии в системе уже были реализованы **non-DP генераторы**
поверх библиотеки SDV: CTGAN, TVAE, CopulaGAN — как baseline для
сравнения качества.

Естественный путь развития: «оборачиваем существующие SDV-генераторы
в DP-механизм» — попытка такого обёртывания была сделана и **не
сработала** (внутреннее устройство SDV-генераторов не позволяет
аккуратно вставить DP-SGD без переписывания их обучения, что в рамках
магистерской работы было бы непропорционально трудозатратно и
методологически рискованно).

После этого встал вопрос: какие готовые DP-инструменты использовать.

## Рассмотренные варианты

### A. SmartNoise Synth (Microsoft / OpenDP)

Открытая Python-библиотека, нативно реализующая DP-CTGAN с
встроенным DP-SGD, privacy accountant'ом (RDP) и privacy-aware
preprocessor для числовых колонок.

* **За:** работает «из коробки»; зрелый privacy accountant; активно
  поддерживается; даёт формальные DP-гарантии без необходимости
  писать DP-SGD самостоятельно.
* **Против:** **не содержит DP-TVAE** — только DP-CTGAN
  (и `dp_marginal_synthesizer`, который по природе другой).
  Для DP-TVAE требуется другое решение.

### B. Opacus (Meta / PyTorch)

Generic-обёртка для DP-SGD над любой PyTorch-моделью:
`PrivacyEngine.make_private_with_epsilon()` навешивает DP на
оптимизатор и DataLoader.

* **За:** работает с любой PyTorch-архитектурой; зрелый и активно
  поддерживаемый; стандарт DP-SGD в PyTorch-экосистеме.
* **Против:** не предоставляет **готовых табличных генераторов** —
  саму архитектуру (encoder/decoder для VAE, generator/discriminator
  для GAN) нужно реализовать самостоятельно или взять из других
  библиотек.

### C. Самостоятельная реализация DP-SGD для всех генераторов

Написать DP-SGD-обёртку поверх PyTorch вручную (с per-sample gradient
clipping, шумом, RDP-учётом).

* **За:** полный контроль; нет внешних зависимостей.
* **Против:** **слишком долго и трудозатратно** для рамок ВКР;
  высокий риск методологических ошибок (некорректный privacy
  accountant — обнуляет смысл DP-гарантий); нет необходимости
  переизобретать то, что уже сделано в Opacus и SmartNoise.

### D. TensorFlow Privacy (TF-Privacy)

DP-SGD от Google поверх TensorFlow.

* **За:** зрелый, поддерживается Google.
* **Против:** требует TensorFlow вместо PyTorch; не было опыта
  работы с TF в проекте; миграция на TF потребовала бы переписывать
  и не-DP генераторы (SDV использует PyTorch); экосистема табличного
  ML на PyTorch плотнее (CTGAN, SDV, snsynth, ctgan).

### E. JAX-Privacy (DeepMind)

DP-механизмы поверх JAX.

* **За:** современный, активно развивается.
* **Против:** требует переход на JAX; ещё больший разрыв с уже
  выбранным PyTorch-стеком; меньше готовых табличных генераторов.

### F. Только Opacus + написать DP-CTGAN самостоятельно

Использовать Opacus для обоих генераторов; DP-CTGAN — переписать
вручную.

* **За:** один DP-фреймворк, единый стиль кода.
* **Против:** ровно та же проблема, что у варианта C, только в
  меньшем масштабе — переписывание CTGAN с DP-SGD-совместимостью
  требует значительных усилий и риска.

### G. Только SmartNoise

Использовать только SmartNoise для DP-генерации.

* **За:** один фреймворк.
* **Против:** SmartNoise не содержит DP-TVAE; пришлось бы
  ограничиться одним типом DP-генератора, что хуже для целей
  диплома (сравнение архитектур — важная часть работы).

## Решение

Принят сочетающийся вариант **A + B**:

* **DP-CTGAN** — реализован поверх **SmartNoise Synth**:
  `final_system/synthesizer/dp_ctgan.py` использует
  `snsynth.Synthesizer` под капотом.
* **DP-TVAE** — реализован поверх **Opacus** + собственная
  PyTorch-архитектура `_VAE` (encoder + decoder, объединённые
  в один `nn.Module` для совместимости с Opacus):
  `final_system/synthesizer/dp_tvae.py` использует
  `opacus.PrivacyEngine.make_private_with_epsilon()`.
* **Не-DP baseline** — CTGAN, TVAE, CopulaGAN на SDV, без изменений.

Аргументация:

1. **Историческая.** SDV-генераторы уже работали к моменту
   принятия решения. Попытка добавить DP-обёртку поверх SDV
   не получилась — вместо этого взят SmartNoise, который
   из коробки даёт DP-CTGAN с корректным privacy accountant'ом.
2. **Покрытие двух архитектурных классов.** SmartNoise покрывает
   DP-CTGAN (GAN-семейство), Opacus — DP-TVAE (VAE-семейство).
   Это даёт два разных DP-генератора для сравнения архитектур
   в одной системе — методологически ценно для ВКР.
3. **Минимизация риска ошибок DP-учёта.** Privacy accountant
   в обеих библиотеках реализован специалистами и проверен; самописный
   accountant — главный источник риска при работе с DP.
4. **Соответствие стеку.** Обе библиотеки — PyTorch-based;
   совмещаются с уже использующимся ctgan/sdv-стеком. TF Privacy
   и JAX-Privacy потребовали бы кросс-фреймворковой миграции
   без оправданной выгоды.
5. **Самостоятельная реализация исключена** как непропорционально
   трудозатратная и рискованная.

## Последствия

### Положительные

* Все 5 генераторов системы (`dpctgan`, `dptvae`, `ctgan`, `tvae`,
  `copulagan`) реализуют единый интерфейс `BaseGenerator`
  (`fit / sample / privacy_report / save / load`) — оркестрация
  Synthesis Service не различает их по типу фреймворка.
* DP-гарантии у обоих DP-генераторов реализованы корректно
  (privacy accountant встроен в SmartNoise; `get_privacy_spent()`
  у Opacus). Это видно в DP-отчёте: `dp_config.is_dp_enabled = true`
  + `dp_spent.spent_epsilon_final` — фактический ε после
  `epochs_completed` эпох.
* Покрыты два разных архитектурных подхода (GAN, VAE) — можно
  сравнивать их между собой и с не-DP baseline'ами в одном
  пайплайне.
* Ни одна строка DP-механизма не написана вручную — снижен риск
  методологических ошибок учёта приватности.

### Отрицательные

* **Два DP-фреймворка** в одном проекте — два разных API
  privacy_report, две разных логики DP-конфига. В коде это
  нормализовано через единый `BaseGenerator.privacy_report()`
  возвращающий одинаковую структуру (`dp_config`, `dp_spent`,
  `dp_guarantees`), но цена нормализации — дополнительный код
  в каждом из двух генераторов.
* **DP-TVAE требует написания и поддержки собственной архитектуры
  `_VAE`** (`encoder + decoder` в одном `nn.Module`). Любые изменения
  библиотеки `ctgan` (откуда берутся базовые слои) могут потребовать
  правок.
* **Зависимость от двух тяжёлых пакетов.** `smartnoise-synth` и
  `opacus` тянут разные подверсии `torch` и могут конфликтовать
  при обновлении версий — образ `synthesis_service` склонен к
  «зацементированным» версиям зависимостей.
* **Версионная привязка SmartNoise.** В коде `dp_ctgan.py` есть
  workaround под конкретный баг SmartNoise 1.0.6 (verbose
  принудительно отключается из-за бага) — возможны подобные
  workaround'ы в будущем.

### Условия пересмотра

Возможные триггеры частичного пересмотра:

* Если в будущем SmartNoise добавит DP-TVAE — рассмотреть отказ
  от Opacus (унификация стека).
* Если SmartNoise или Opacus прекратят поддержку — искать замену
  для соответствующего генератора.
* Если потребуется DP для генератора, не поддерживаемого ни
  SmartNoise, ни Opacus (например, DP-CopulaGAN, DP-Diffusion) —
  скорее всего использовать Opacus (как generic-обёртку).
* Появление single-stack DP-tabular toolbox'а с поддержкой обоих
  семейств моделей — рассмотреть миграцию для упрощения.
