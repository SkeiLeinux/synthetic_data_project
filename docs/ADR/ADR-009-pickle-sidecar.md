# ADR-009 — Pickle + JSON-сайдкар для сохранённых моделей

**Статус:** Принято (временное)
**Дата:** 2026-04-25
**Связанные ADR:** [ADR-001](ADR-001-microservices.md), [ADR-012](ADR-012-lazy-ml-imports.md)

## Контекст

При `save_model=true` Synthesis Service после обучения должен
сохранить генератор так, чтобы:

1. **Synthesis Service** мог в будущем `load()` модель и выполнить
   `sample()` без повторного обучения (`POST /models/{id}/sample`,
   ε-бюджет не расходуется — post-processing immunity DP).
2. **Gateway** мог отдавать пользователю метаданные модели
   (`GET /models`, `GET /models/{id}`): `dp_config`, `dp_spent`,
   `dataset_name`, `created_at`, `epochs_completed`, и т.д.

Природа сохраняемых объектов:

* Это полноценные Python-объекты с внутренним состоянием:
  `DPCTGANGenerator` оборачивает `snsynth.Synthesizer` со встроенными
  data transformers; `DPTVAEGenerator` — обёртка вокруг PyTorch-модели
  с `ctgan.DataTransformer`; SDV-генераторы — собственные классы.
* Внутренние трансформеры этих библиотек (категориальные encoder'ы,
  спектральные нормализации, обученные state'ы) **не имеют
  стандартизованной сериализации** в общеотраслевые форматы
  (ONNX/safetensors).

Дополнительное архитектурное ограничение:

* Gateway по соображениям [ADR-012](ADR-012-lazy-ml-imports.md)
  **не должен импортировать ML-классы**, поэтому он не может
  выполнить `pickle.load` модели — иначе ему понадобятся
  `synthesizer/`, `torch`, `opacus`, `smartnoise-synth` в образе.

## Рассмотренные варианты

### A. Только pickle (`{model_id}.pkl`)

Один pickle-файл; Gateway вынужден либо загружать его (=притянуть
все ML-зависимости), либо отдавать без метаданных.

* **За:** одна точка хранения.
* **Против:** нарушает [ADR-012](ADR-012-lazy-ml-imports.md);
  либо Gateway тяжёлый, либо у него нет метаданных модели.

### B. Pickle + JSON-сайдкар (выбранный вариант)

Два файла: `{model_id}.pkl` (модель) + `{model_id}.meta.json`
(метаданные). Synthesis Service пишет оба; Gateway читает только
JSON-сайдкар через `_load_sidecar()`.

* **За:** метаданные доступны Gateway без `pickle.load` и без
  ML-зависимостей; разделение «тяжёлая модель / лёгкие метаданные».
* **Против:** два файла надо держать консистентными; pickle всё
  равно небезопасен и привязан к версиям библиотек.

### C. Метаданные внутри pickle + дублирование в БД

Метаданные внутри pickle (`payload["metadata"]`) + дублирующая
запись в PostgreSQL (`models`-таблица), которую читает Gateway.

* **За:** Gateway не открывает pickle, читает БД.
* **Против:** усложнение схемы БД; новое разделение «БД vs FS»
  для одной сущности; миграция БД при изменении полей метаданных.

### D. ONNX / safetensors для весов + JSON для метаданных

Полная замена pickle на безопасный формат.

* **За:** безопасно (ONNX/safetensors не выполняют код при
  загрузке); версии-независимо; стандартный формат.
* **Против:** ни SmartNoise, ни SDV не сохраняют свои генераторы
  в ONNX «из коробки»; пришлось бы вручную выгружать веса PyTorch-
  моделей и **отдельно** сохранять состояние data transformers
  (категориальные encoder'ы, бинаризаторы) — нетривиальная работа,
  выходящая за scope MVP.

### E. Pickle + protobuf-схема для метаданных

То же, что B, но метаданные в protobuf.

* **За:** строгий мульти-язычный контракт.
* **Против:** добавляет proto-зависимость без выгоды для Python-only
  системы (см. [ADR-005](ADR-005-pydantic.md)).

## Решение

Принят вариант **B — pickle для модели + JSON-сайдкар для метаданных**.

Реализация:

* **Synthesis Service** (`services/synthesis_service/router.py:_write_model_sidecar`)
  при `save_model=true` после `generator.save(pkl_path)` записывает
  параллельный `{model_id}.meta.json` со всеми полями, нужными Gateway:

  ```json
  {
    "model_id": "...",
    "run_id": "...",
    "dataset_name": "...",
    "generator_type": "dpctgan",
    "created_at": "2026-04-25T10:30:15Z",
    "file_size_bytes": 12345678,
    "privacy_report": {
      "dp_config": { "is_dp_enabled": true, "epsilon_initial": 3.0, ... },
      "dp_spent":  { "spent_epsilon_final": 2.48, "epochs_completed": 87, ... },
      "data":      { "sample_size": 32561 }
    }
  }
  ```

* **Gateway** (`api/routers/models.py:_load_sidecar`) читает только
  JSON-сайдкар. Если файл отсутствует или повреждён — метаданные
  считаются недоступными, но `model_id` и базовые поля (created_at,
  file_size_bytes) восстанавливаются из FS-stat'а самого pkl-файла.

* **Sample из модели** — единственная операция, требующая `pickle.load`,
  выполняется внутри Synthesis Service (`POST /models/{id}/sample`).
  Gateway проксирует запрос, не открывая pickle сам.

Аргументация:

1. **Pickle проще**, чем поэлементное сохранение весов и состояний
   data transformers SmartNoise/CTGAN — особенно с учётом, что эти
   библиотеки не предоставляют стандартизованной сериализации.
2. **Pickle лучше работает с этими конкретными библиотеками**:
   `snsynth.Synthesizer` восстанавливается через pickle практически
   гарантированно той же версией; экспериментальные попытки иной
   сериализации требовали бы ручного отвязывания внутренних state'ов.
3. **Сайдкар** разрывает связь между Gateway и pickle — Gateway
   остаётся слим (см. [ADR-012](ADR-012-lazy-ml-imports.md))
   и не зависит от наличия классов синтезаторов.
4. Это **временное** решение: ограничения pickle в plain enterprise
   признаны (RCE-риск, версия-зависимость), и переход на
   safetensors+ONNX зафиксирован в PRD раздел 12.4.3 как явный
   технический долг.

## Последствия

### Положительные

* `Gateway` остаётся слим: не содержит `torch`/`opacus`/`smartnoise`,
  не выполняет `pickle.load`.
* Метаданные модели доступны через `GET /models/{id}` без поднятия
  Synthesis Service.
* Удаление модели (`DELETE /models/{id}`) удаляет оба файла
  (pkl + sidecar) атомарно с точки зрения пользовательского API.
* Если в будущем потребуется добавить новое поле метаданных
  (например, hash обучающего split'а) — это правится только
  в `_write_model_sidecar` и схеме `ModelDetail`, без миграций
  существующих моделей.

### Отрицательные

* **Pickle небезопасен.** `pickle.load` выполняет произвольный код
  при загрузке. Сейчас pickle-файлы лежат на доверенном volume
  `/data/models`, но при production-внедрении это блокер
  (PRD 12.4.3).
* **Версионная привязка.** Если обновить версию `snsynth`/`ctgan`/`torch`,
  ранее сохранённые модели могут перестать загружаться. Нет
  явной маркировки «модель X сохранена с версиями Y» внутри pickle —
  можно добавить в sidecar в будущем.
* **Два файла надо держать консистентными.** Если `_write_model_sidecar`
  упал после `generator.save()` — есть pickle без метаданных.
  Текущая обработка — graceful: Gateway вернёт `dp_config: null`
  и базовые поля из stat'а файла.
* Нет автоматической проверки целостности (хеша) pickle-файла
  при загрузке.

### Условия пересмотра

Перейти на безопасный формат (PRD 12.4.3) при наступлении любого из:

* Production-внедрение, где модели могут попасть из недоверенных
  источников (multi-tenant, скачивание из внешних реестров).
* Интеграция с MLflow Model Registry, который требует стандартный
  формат.
* Появление требования cross-version compatibility (модель,
  обученная год назад, должна сэмплировать сейчас).

Кандидаты на замену:

* **safetensors** для PyTorch-весов (DP-TVAE) + JSON для состояний
  data transformers — потребует ручной выгрузки и сборки.
* **ONNX** для inference-graph — сложнее с custom-слоями SmartNoise.
* В случае невозможности отказа — добавить хотя бы HMAC-подпись
  pickle-файлов и проверку перед загрузкой.
