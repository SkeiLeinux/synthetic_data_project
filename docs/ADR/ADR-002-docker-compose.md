# ADR-002 — Docker Compose как платформа развёртывания

**Статус:** Принято (для MVP)
**Дата:** 2026-04-25
**Связанные ADR:** [ADR-001](ADR-001-microservices.md), [ADR-003](ADR-003-shared-volume.md)

## Контекст

После принятия решения о микросервисной архитектуре (см.
[ADR-001](ADR-001-microservices.md)) встал вопрос, чем именно
оркестрировать 5 контейнеров приложения + Redis + 2 PostgreSQL.

Задача формулировалась в контексте магистерской работы:

* Сценарий запуска — **локальная машина или одна виртуальная машина**;
  весь стек поднимается одной командой.
* Аудитория «первого запуска» — рецензенты ВКР, разработчики,
  потенциальные пользователи enterprise-уровня, которые хотят
  попробовать систему перед полноценной интеграцией.
* Нет требований к horizontal scaling, rolling updates, multi-tenancy
  на этапе MVP.
* Нужен GPU-доступ для одного контейнера (`synthesis_service`).

## Рассмотренные варианты

### A. `docker run` руками + bash-скрипт

Запускать каждый контейнер отдельной командой `docker run` с правильным
networking и volumes; обернуть в bash-скрипт.

* **За:** ноль зависимостей сверх Docker.
* **Против:** длинные команды, сложно поддерживать, нет healthcheck'ов
  и зависимостей между контейнерами «из коробки».

### B. Docker Compose

Один `docker-compose.yml` описывает все сервисы, сети, volumes,
healthcheck'и, depends_on.

* **За:** стандарт де-факто для локальной мульти-контейнерной разработки;
  декларативно; включает healthcheck'и и `depends_on: condition: service_healthy`;
  нативная поддержка GPU (`deploy.resources.reservations.devices`).
* **Против:** single-host; нет автомасштабирования; нет rolling updates;
  при production-нагрузке упрётся в один хост.

### C. Kubernetes (k3s / kind / minikube для локалки + полноценный для прод)

Helm-чарты, манифесты, ingress.

* **За:** production-grade с первого дня; horizontal scaling,
  rolling updates, namespaces, network policies.
* **Против:** существенная избыточность для MVP уровня магистерской ВКР;
  кратно более высокий порог входа для рецензента «просто запустить»;
  отдельно установить kubectl, helm, локальный k8s-кластер; настроить
  GPU-плагин — всё это лишние шаги для академической демонстрации.

### D. HashiCorp Nomad / Docker Swarm

Промежуточные оркестраторы.

* **За:** проще K8s, мощнее Compose.
* **Против:** не дают принципиально новых возможностей по сравнению с Compose
  для single-host MVP, но добавляют установочные шаги.

## Решение

Принят вариант **B — Docker Compose**.

Аргументация:

1. Соответствует scope MVP — single-host, локальный запуск, академическая
   демонстрация.
2. `docker-compose.yml` декларативно описывает все 8 контейнеров
   (Gateway + 4 микросервиса + Redis + 2 PostgreSQL), включая
   healthcheck'и и зависимости — этого достаточно для надёжного
   подъёма всего стека одной командой.
3. Нативная поддержка GPU через `deploy.resources.reservations.devices`
   позволяет передать GPU только в `synthesis_service`, не вводя
   зависимостей в остальные.
4. Для рецензента ВКР путь «`cp .env.example .env` → `docker compose up -d --build`»
   укладывается в две команды — это минимальный возможный порог входа.

Production-внедрение в крупной компании требует перехода на
Kubernetes (см. PRD, раздел 12.1.4) — это вынесено в отдельный пункт
roadmap и **не отрицает** решение текущего ADR, а является следующим
шагом эволюции.

## Последствия

### Положительные

* Один файл `docker-compose.yml` — единственный источник правды
  по конфигурации развёртывания.
* `docker compose up -d --build` поднимает весь стек; healthcheck'и
  обеспечивают корректный порядок старта.
* GPU прокидывается только в `synthesis_service`, не требуется на хосте
  для остальных сервисов (которые могут запускаться даже без GPU).
* Простой откат (`docker compose down`).
* Легко интегрируется в e2e-тест: `pytest -m e2e` поднимает стек
  через `docker compose up -d --wait` (см. `tests/test_e2e_microservices.py`).

### Отрицательные

* **Single-host.** Все 8 контейнеров делят ресурсы одной машины —
  упор в CPU/RAM/GPU при росте нагрузки.
* **Нет автомасштабирования.** Нельзя поднять вторую реплику
  Synthesis Service для параллельных джобов.
* **Нет rolling updates.** Деплой новой версии = полное пересоздание
  контейнера и кратковременный downtime.
* **Bind-mount `/data` не подходит для multi-host.** Шаг к multi-host
  потребует одновременной замены volume на сетевое хранилище
  (см. [ADR-003](ADR-003-shared-volume.md)) и переезда на другой
  оркестратор.

### Условия пересмотра

Перейти на Kubernetes (с Helm-чартами) при наступлении любого из:

* Появление требования multi-host деплоя (нагрузка не помещается
  в один узел).
* Появление требования автомасштабирования (например, очередь
  пайплайнов растёт быстрее, чем один Synthesis Service её разгребает).
* Внедрение в корпоративное окружение, где K8s — стандарт,
  а Docker Compose — нет.
* Появление требования zero-downtime deploy.
