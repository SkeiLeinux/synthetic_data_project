# ADR-003 — Shared Docker volume `/data` для передачи артефактов

**Статус:** Принято (для MVP)
**Дата:** 2026-04-25
**Связанные ADR:** [ADR-001](ADR-001-microservices.md), [ADR-002](ADR-002-docker-compose.md)

## Контекст

В пайплайне «импорт → split → синтез → оценка → отчёт» каждый шаг
производит файлы (CSV, JSON, pickle), которые потребляются следующими
шагами:

* `splits/{split_id}/{train,holdout}.csv` — пишет Data Service,
  читают Synthesis и Evaluation;
* `synth/{job_id}/synthetic.csv` — пишет Synthesis, читает Evaluation;
* `models/{model_id}.{pkl,meta.json}` — пишет Synthesis, читает Gateway;
* `reports/{dataset}__{generator}__{ts}.json` — пишет Reporting.

Размер артефактов:

* CSV-датасеты — десятки МБ типично, до 100–500 МБ на верхнем краю;
* модели после `pickle.dump` — десятки МБ;
* отчёты — десятки–сотни КБ.

Нужен механизм межсервисного обмена этими артефактами.

## Рассмотренные варианты

### A. Передача через тело HTTP-запроса (multipart / streaming)

Каждый сервис принимает и возвращает файлы прямо в теле запроса.

* **За:** stateless, нет shared state, легко масштабируется
  по разным хостам.
* **Против:** один и тот же CSV в 100 МБ ходит по сети 3–4 раза
  (Data → Synth, Data → Eval, Synth → Eval); накладные расходы на
  сериализацию/копирование; синтетика после генерации физически
  больше, чем исходный train.

### B. Object storage (S3 / MinIO / Ceph RGW)

Каждый сервис читает/пишет напрямую в S3-совместимое хранилище;
между собой передают только ссылки (URI).

* **За:** production-grade; multi-host friendly; versioning, lifecycle
  policies, шифрование at rest «из коробки» у managed-сервисов.
* **Против:** требует поднимать MinIO/Ceph (или зависеть от внешнего S3)
  даже для MVP-запуска; нужен абстрактный слой `Storage` в коде;
  дополнительная конфигурация креденшалов.

### C. Очередь сообщений с inline payload (Kafka / RabbitMQ)

Артефакты передаются как сообщения через брокер.

* **За:** event-driven, durable очередь, retry семантика.
* **Против:** размер сообщений исчисляется десятками МБ — выходит
  за typical sweet spot обоих брокеров; всё равно потребуется
  отдельное хранилище для артефактов.

### D. Shared Docker volume

Один `volume: shared_data` смонтирован во все 4 микросервиса
по пути `/data`. Сервисы обмениваются **относительными путями**
(`splits/{id}/train.csv`), а не самими файлами.

* **За:** нулевые накладные расходы на передачу; никакой сериализации/
  десериализации; один артефакт записывается один раз и читается
  напрямую с локальной FS.
* **Против:** single-host (Docker bind-mount не работает между хостами);
  нет встроенного versioning; конкурентная запись опасна (что вынуждает
  держать `--workers 1`).

## Решение

Принят вариант **D — shared Docker volume `/data`**.

Структура:

```
/data/
├── datasets/{dataset_id}/{raw.csv, meta.json}
├── splits/{split_id}/{train.csv, holdout.csv, profile.json, meta.json}
├── synth/{job_id}/{synthetic.csv}              # после атомарного rename
├── models/{model_id}.pkl + {model_id}.meta.json
├── reports/{dataset}__{generator}__{ts}.json
└── logs/{service}.log
```

Конвенция:

* Сервисы передают друг другу **только относительные пути**
  (например, `splits/abc-123/train.csv`), а не сами файлы.
* Корень разделяемого тома доступен через переменную окружения
  `DATA_ROOT` (по умолчанию `/data`).
* Финальные артефакты записываются атомарно: для синтетики — через
  pattern `synthetic_pending.csv` → `synthetic.csv` (rename атомарен на FS).

Аргументация:

1. На MVP-уровне нет требования multi-host, поэтому single-host —
   приемлемое ограничение.
2. Артефакты не передаются по сети 3–4 раза, что критично для
   крупных датасетов.
3. Решение не требует поднимать дополнительный сервис (MinIO/RabbitMQ),
   что соответствует общему принципу «минимум инфраструктуры
   для запуска».
4. Переезд на S3 в production не требует переписывания контрактов
   между сервисами — нужно только заменить реализацию `Storage`,
   так как в API уже передаются относительные пути, а не байты файлов.

## Последствия

### Положительные

* Тяжёлые артефакты (CSV-датасеты, синтетика, модели) не ходят по сети;
  читаются напрямую с локальной FS.
* HTTP-контракты между сервисами компактные (передаются только пути
  и метаданные).
* Простота отладки: можно `docker compose exec` зайти в любой
  контейнер и `ls /data/splits/...`.

### Отрицательные

* **Single-host.** Docker bind-mount не работает между нодами.
  Это вместе с [ADR-002](ADR-002-docker-compose.md) фиксирует
  один из главных гэпов на пути к production.
* **Нет встроенного versioning** артефактов — повторная загрузка
  датасета с тем же `dataset_id` затрёт raw.csv.
* **Опасность конкурентной записи.** Несколько worker'ов одного
  сервиса могут одновременно писать в одну папку — поэтому все
  сервисы запускаются с `--workers 1`. Это технический долг.
* **Нет шифрования at rest** — данные лежат plaintext на хост-FS.

### Условия пересмотра

Перейти на S3-совместимое хранилище (см. PRD, раздел 12.1.3) при
наступлении любого из:

* Деплой требует multi-host (вместе с переездом на K8s).
* Появление требования versioning артефактов.
* Появление требования шифрования at rest или интеграции с KMS.
* Появление требования multi-tenant изоляции (per-tenant prefix в S3).

При переходе нужно:

1. Ввести абстрактный интерфейс `Storage` (Strategy pattern).
2. Реализовать `LocalFSStorage` (текущее поведение) и `S3Storage`.
3. Заменить пути типа `splits/{id}/train.csv` на pre-signed URL
   или сохранить относительные пути и резолвить их через storage.
