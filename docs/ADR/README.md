# Architecture Decision Records (ADR)

В этой директории хранятся записи об архитектурных решениях
(Architecture Decision Records, ADR), принятых при проектировании
системы генерации синтетических табличных данных с дифференциальной
приватностью.

## Зачем нужны ADR

ADR фиксируют **причины** архитектурных выборов в момент их принятия.
Через год / при смене команды / при ревью «почему так сделано»
ответ находится в одном месте, а не реконструируется из git-истории
и архивных переписок.

Каждая запись отвечает на четыре вопроса:

1. **Контекст** — какая была проблема, какие были ограничения.
2. **Альтернативы** — что ещё рассматривалось.
3. **Решение** — что выбрано и почему именно оно.
4. **Последствия** — что это нам даёт и что мы теряем; при каких
   условиях решение нужно пересмотреть.

## Формат

Используется лёгкая адаптация **MADR (Markdown ADR)**:

```
# ADR-NNN — Краткий заголовок
**Статус:** Принято | Предложено | Отвергнуто | Заменено ADR-XXX
**Дата:** YYYY-MM-DD

## Контекст
## Рассмотренные варианты
## Решение
## Последствия
  ### Положительные
  ### Отрицательные
  ### Условия пересмотра
## Связанные ADR
```

ADR неизменяемы после принятия. При смене подхода создаётся новый ADR
со ссылкой `Supersedes ADR-XXX` / `Superseded by ADR-YYY`.

## Когда писать новый ADR

Пишите новый ADR, если:

* меняете технологию, которая повлияет на несколько сервисов
  (БД, очередь, фреймворк, формат сериализации);
* делаете методологический выбор, который нельзя восстановить
  из кода (например, новый тип атаки на приватность);
* отказываетесь от уже принятого ADR — тогда новый ADR должен
  явно указать `Supersedes ADR-XXX` и зафиксировать причины.

Не пишите ADR для:

* мелких рефакторингов и переименований;
* выборов внутри одного модуля без архитектурных последствий;
* решений, которые уже задокументированы в коде через docstrings.
